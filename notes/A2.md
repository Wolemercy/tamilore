## Concept in my own words

- A connection (port) is created after a 3-Way TCP handshake on the listening port.
- `socket.listen(int backlog)` tells the server to listen for connections to that port. The backlog is the number of connections that can be queued up. If it's too small, then a valid connections can get dropped/fail if the server receives more than is allowed before dequeing them. If it's too large, then the server could end up with more connections than it can handle/accept, giving false hope to clients
- When a connection is established after `accept()`, the server gets a new file descriptor for communicating with the client. The connection is identified using the `protocol, server_address:port, client_address:port`, and each unique combination has its file descriptor.

## Mastery check

**Q1:** What is the backlog queue?
**A:** The number of connections that can be queued up, pending `accept()`ing.

**Q2:** What happens to connection #6 if `backlog=5` and you haven't called `accept()` yet?
**A:** If `tcp_abort_on_overflow=0`, the server drops the `SYN` and after the client retries with exponential backoff, it gives up with a timeout error. If `tcp_abort_on_overflow=1`, the kernel sends an RST(Reset) packet and the client sees the ConnectionRefusedError.

**Q3:** Trace the kernel-level flow when a client connects.
**A:** Client kernel picks an ephemeral port for ths socket, then sends SYN to server at `address, port`. Server Kernel receives SYN, and checks if `(protocol, address, port)` exists in its socket table. If there is no listening socket, then a `ConnectionRefusedError` is raised by the client. If there is, it sends SYN-ACK to client. Client kernel sends ACK to Server Kernel, and a connection is established. Connection is moved into the backlog accept queue. `accept()` dequeues the connection from the backlog, and kernel assigns a new file descriptor for the connection socket, and communication happens as a result.
