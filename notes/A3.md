# A3 — Send and Receive Byte Streams

**Prerequisites:** A2
**Status:** In Progress

## Build task (from curriculum)

Build a server that accepts a connection and calls `recv(1024)`. Build a client that sends a 4096-byte message. Print how many bytes the server actually received on the first `recv()`. Run it 10 times. Observe that the number varies.

**Read:** Beej's Guide, section 6.

## Concept in my own words

TCP is a byte stream. Bytes go from sender app -> sender kernel/send buffer -> network line -> receiver kernel/receive buffer -> receiver app via recv(). When bytes are sent, they are guaranteed to arrive in order. However, some may be stuck at any point in the path described above. So a single `recv(1024)` may not return 1024 bytes even if the client sent `>=1024` bytes.

In a scenario where the client sends this

```
s.send("Hello")
s.send("Tamilore")
```

The server may receive any or more of the following

```
-> "HelloTamilore"
-> "Hello" then "Tamilore"
-> "He" then "lloTamilore"
```

## Build summary

Accross the 10 `recv(1024)` runs, the server consistently returned 1024 bytes. It's difficult to simulate that it can receive less than 1024 bytes on localhost.

## Mastery check

**Q1:** Why does `recv(1024)` sometimes return fewer than 1024 bytes even when the client sent more?
**A:** Because some bytes may not have arrived when `recv` is called.

**Q2:** Where are the remaining bytes?
**A:** Anywhere in the path from the sender to the receiver (sender kernel/send buffer, network line, receiver kernel/receiver buffer)

**Q3:** What guarantees does TCP actually provide?
**A:** It guarantees that bytes would be received in the order in which they were sent.
